## Problem Statement

Claude Code 在后台跑任务时经常需要用户手动确认（权限、工具调用等），但用户切走页面后看不到提示，导致任务卡住。用户不想开狂暴模式（权限太大），也不想一直盯着终端。需要一个持续可见的信号来提醒用户。

## What Makes This Cool

红绿灯不是通知——通知弹一下就消失，红绿灯一直在那亮着。你切回浏览器、回微信、回飞书，扫一眼菜单栏就知道状态。这是通知做不到的。加上 context 阈值监控，这是一个真正的 Claude Code "驾驶仪表盘"。

## Constraints

- macOS 原生菜单栏应用，纯 Swift/SwiftUI
- 不依赖 Claude Code Hooks（解耦，Hooks 作为可选增强）
- 零配置可用——装上就能用，不需要改 Claude Code 设置
- 开源项目，其他 Claude Code 用户也能用
- 支持多会话（同时跑多个 Claude Code 实例）

## Premises

1. **数据源是进程状态 + 本地文件轮询，不是 Hooks** — 进程状态检测（waiting on stdin）判断是否在等确认，本地文件轮询判断 context 使用量。Hooks 可选增强，但不依赖。
2. **纯 Swift 菜单栏应用是正确载体** — 体积小（几 MB）、无外部依赖、原生体验、分发简单（.dmg / .zip）。
3. **核心价值是持续可见的状态信号** — 红绿灯模式 vs 通知模式的本质区别：信号持续存在，不需要用户主动查看。

## Approaches Considered

### Approach A: 极简版
Swift 菜单栏 App，只做红绿灯 + 点击跳转终端。~500 行代码，1-2 天。

### Approach B: 标准版
A 基础上加 context 阈值监控 + 多会话 + 声音提醒 + Hooks 可选增强。~1500 行代码，3-5 天。

### Approach C: 完整版（选中）
B 基础上加 Popover 窗口（会话详情、context 用量百分比）+ LaunchAtLogin。~3000 行代码，1-2 周。

## Recommended Approach

完整版，分阶段交付。不叫"方案 C"，直接说做什么：

- **Phase 1**：进程监控 + context 阈值 + 多会话 + 声音 + 点击跳转
- **Phase 2**：Popover 窗口（会话列表、context 进度条；布局 wireframe 在 Phase 2 开始前设计）
- **Phase 3**：阈值配置 + LaunchAtLogin

### 信号源设计

**Spike 验证结果（2026-06-08）：**

`~/.claude/sessions/{pid}.json` 包含实时会话状态：
```json
{
  "pid": 5126,
  "sessionId": "a3d5e893-9661-4cab-87e6-c1f9bcdb2c1d",
  "status": "busy",        // "busy" | "idle" | "waiting" — 直接可用！
  "waitingFor": "permission prompt",  // 仅 status="waiting" 时出现，说明等待原因
  "cwd": "/Users/lyle/Desktop/Projects",
  "startedAt": 1780886988727,
  "procStart": "Mon Jun  8 02:49:48 2026",
  "version": "2.1.168",
  "kind": "interactive",
  "entrypoint": "cli",
  "updatedAt": 1780889438380
}
```

`~/.claude/projects/{slug}/{session-id}.jsonl` 中 assistant 消息含 `usage` 字段（嵌套在 `message.usage` 下）：
```json
{
  "type": "assistant",
  "message": {
    "usage": {
      "input_tokens": 31851,
      "cache_read_input_tokens": 22208,
      "output_tokens": 5
    }
  }
}
```

| 信号源 | 检测方式 | 频率 | 状态映射 | 可靠性 |
|--------|----------|------|----------|--------|
| 会话状态（主信号） | 读取 `~/.claude/sessions/{pid}.json` 的 `status` 字段 | 2s | `"busy"` → running/绿灯；`"idle"` → idle/灰灯；`"waiting"` → confirming/红灯 | ✅ Spike 验证通过 |
| 等待原因（主信号增强） | 读取同一文件的 `waitingFor` 字段（仅 `status="waiting"` 时存在） | 2s | `"permission prompt"` → 菜单/Poper 显示等待原因 | ✅ Review 验证通过 |
| 进程检测（兜底） | `ps -o pid,stat,args` 轮询 claude 进程，过滤仅匹配 CLI（`comm == "node"` 且 `args` 包含 `@anthropic-ai/claude-code`），排除 Claude Desktop | 5s | 检测到进程但无 session 文件 → 用进程状态推断 | ✅ 兜底方案 |
| Context 大小 | 读取 `~/.claude/projects/{slug}/{session-id}.jsonl` 全量，取末尾 `type: "assistant"` 消息的 `message.usage` 字段 | 5s | input_tokens + cache_read > 150K → 黄灯, > 200K → 红灯 | ✅ Spike 验证通过 |
| Hooks (可选) | PreToolUse 脚本写入 `~/.claude/signal-hook-state.json` | 事件驱动 | 等待确认 → 红灯 + 原因 | ✅ 可靠（但非必需） |

### 状态模型

```
enum SignalState {
    case idle        // 无 claude 进程，或 session.status = "idle" → 灰灯 (circle)
    case running     // session.status = "busy" → 绿灯 (play.fill)
    case confirming  // session.status = "waiting" → 红灯（1Hz 闪烁, exclamationmark.triangle.fill）
    case warning     // context tokens > 150K → 黄灯（0.5Hz 闪烁, exclamationmark.circle.fill）
    case critical    // context tokens > 200K → 红灯（常亮, xmark.circle.fill）
    case error       // 检测失败 → 灰灯 + 菜单内提示错误 (xmark.octagon)
}
```

**颜色与无障碍：** 灯色通过 SF Symbol 图标形状区分（见 enum 注释），不依赖纯色区分。闪烁频率：红灯 1Hz，黄灯 0.5Hz，常亮状态不闪烁。

**空状态 vs 错误状态视觉区分：** 无会话时显示低透明度灰灯（circle），error 状态显示 xmark.octagon。两者图标形状不同 + 透明度不同，用户一眼区分"正常空状态"和"App 出错"。

**多会话状态聚合优先级：** confirming > critical > warning > running > idle > error。多个会话同时存在时，取优先级最高的状态作为全局图标状态。confirming 和 critical 都是红灯，但 confirming 优先（需要用户操作 vs 只需关注）。

**误报控制（核心 UX 问题）：** Spike 验证发现 `sessions/{pid}.json` 的 `status` 字段直接区分 `"busy"` / `"idle"` / `"waiting"`，不存在之前担心的 `S+` 状态误判问题。进程检测降级为兜底方案。

**进程退出处理：** 进程消失后保持最后状态 5 秒，然后转 idle。5 秒内新进程出现（同 working directory）则视为同一会话。

**首运行体验：** 无 claude 进程时显示灰灯 + 提示 "No Claude Code sessions detected"。无 `~/.claude/` 目录时提示 "Claude Code not found"。

**声音提醒：** 默认使用系统音效 `Sosumi`（macOS 内置），等待状态首次进入时响一次，不重复。每会话独立跟踪"首次进入"冷却——同一会话不会重复响铃，不同会话进入 confirming 各响一次。可通过菜单项静音。

### 多会话支持

- 每检测到一个 `claude` 子进程 = 一个会话
- 会话标识：working directory（主标识）+ PID（辅助标识），格式 `~/Projects/my-app (PID 12345)`
- 每个会话独立跟踪状态和 context
- 菜单栏图标显示最严重的状态（任一红灯 → 整体红灯）
- Popover 中列出所有会话及其独立状态

### Context 监控实现

**Spike 验证通过（2026-06-08）：** `~/.claude/projects/{slug}/{session-id}.jsonl` 中每条 assistant 消息包含 `usage` 字段（`input_tokens` + `cache_read_input_tokens` + `output_tokens`）。读取 jsonl 文件末尾的 assistant 消息即可获得实时 token 数。

**实现要点：**
- 通过 `sessions/{pid}.json` 的 `sessionId` 字段关联到 `projects/{slug}/{sessionId}.jsonl`
- 读取 jsonl 中 `type: "assistant"` 消息的 `message.usage` 字段（注意：usage 嵌套在 message 对象内，不在顶层）
- 总 token 数 = `input_tokens` + `cache_read_input_tokens`（**不含** `output_tokens`，因为 output 是模型生成的，不消耗 context window）
- **usage 缺失处理**：如果 `message.usage` 字段不存在，保持上次已知值，菜单/Poper 显示 "usage data unavailable"。不置 0（避免误导用户以为 context 安全）
- **僵尸 session 处理**：每次轮询时用 `kill(pid, 0)` 验证进程是否存活。不存在的进程标记为 stale，5 秒后转 idle
- **cwd-slug 映射容错**：如果 `projects/{slug}/` 目录不存在（非项目目录启动的会话），context 值为 0，不报错

### Hooks 可选增强

用户配置 PreToolUse Hook 脚本，将等待原因写入约定路径：
```json
// ~/.claude/signal-hook-state.json
{
  "session_id": "abc123",
  "state": "waiting",
  "reason": "Bash: rm -rf /tmp/test",
  "since": "2026-06-08T11:00:00Z"
}
```
App 读取此文件获取更丰富的等待原因信息。不配 Hook 就没有这一层，核心功能不受影响。

## Open Questions

1. ~~**`~/.claude/` 目录结构**~~ — ✅ Spike 已验证，session 文件和 jsonl 均可用
2. ~~**macOS 进程状态检测精度**~~ — ✅ 不再是主方案，`sessions/{pid}.json` 的 `status` 字段更精确，`ps` 降级为兜底
3. **终端窗口跳转** — Terminal.app ✅、iTerm2 ✅、Warp ❌（不支持 AppleScript）。tmux/screen fallback 为复制命令到剪贴板。
4. **AppleScript Accessibility 权限** — 首次使用跳转功能时检测 Accessibility 权限。未授权时菜单项显示"需要授权"并引导用户到系统偏好设置。未授权不阻塞其他功能。
5. **轮询 vs FSEvents** — Phase 1 用 Timer 轮询先交付。FSEvents（DispatchSource.makeFileSystemObjectSource）可实现零 CPU 空闲 + 即时通知，Phase 3 考虑迁移。
6. **LSUIElement App 内存压力** — macOS 在内存压力下可能杀死无 Dock 图标的 App。Phase 3 加 LaunchAgent keepAlive 或 crash 检测 + 自动重启。
7. **子代理会话** — Claude Code 子代理（subagents/）有独立的 context 使用，Phase 1 不监控子代理，后续版本扩展。

## Success Criteria

1. 安装后无需任何配置，启动 App 即可看到 Claude Code 进程状态
2. 从检测到状态变化到 UI 更新 < 2 秒（轮询间隔 2s，最坏情况 ~4 秒从实际状态变化到可见）
3. Context 达到阈值时黄灯/红灯提醒
4. 支持同时监控 3+ 个 Claude Code 会话
5. 点击菜单栏图标可跳转到对应终端窗口（Terminal.app、iTerm2、Warp；tmux/screen fallback 复制命令到剪贴板）
6. Popover 显示每个会话的 context 用量百分比（Phase 2）

## 视觉设计规范（DevEx Review 2026-06-08）

参考 Apple Human Interface Guidelines 和 macOS 控制中心设计语言。

### 菜单栏图标

| 状态 | SF Symbol | 颜色 | 动画 | 备注 |
|------|-----------|------|------|------|
| idle | `circle` | Color.gray.opacity(0.4) | 无 | 低透明度，区分 error |
| running | `play.fill` | Color.green | 无 | 稳定绿灯 |
| confirming | `exclamationmark.triangle.fill` | Color.red | `.symbolEffect(.pulse)` | 呼吸式脉冲，非传统闪烁 |
| warning | `exclamationmark.circle.fill` | Color.yellow | `.symbolEffect(.variableColor)` | 渐变式效果 |
| critical | `xmark.circle.fill` | Color.red | 无 | 常亮红灯，不闪烁 |
| error | `xmark.octagon` | Color.gray | 无 | 区别于 idle（不同图标形状） |

**图标风格**：多色 SF Symbol（用户优先版），形状 + 颜色双重编码。不使用 template 模式——核心价值是扫一眼看颜色。

**颜色系统**：默认用 Apple 系统语义色（`Color.green`/`.red`/`.yellow`/`.gray`），自动适配暗色/亮色模式。Phase 3 加用户自定义颜色选项。

**macOS 版本兼容**：`.symbolEffect(.pulse)` 需要 macOS 14+（Sonoma）。低版本 fallback 到 Timer 驱动的图标切换动画。

### 状态转换动画

```swift
withAnimation(.spring(response: 0.35, dampingFraction: 0.8)) {
    signalState = newState
}
```

- 所有状态变化使用 Spring 弹性动画（Apple 标准交互曲线）
- SF Symbol 变体切换自动支持 content transition 动画
- 图标变形有弹性感，颜色渐变丝滑
- 硬切（无动画）禁止——让 App 看起来廉价

- 高度：6pt
- 右侧显示数值（如 "142K / 200K"），`.caption` 字体

### 声音设计

- 默认音效：`NSSound(named: "Sosumi")` — macOS 经典音效，辨识度高
- Phase 3 可加自定义音效选项

### 首次体验设计

1. **启动后 0 秒**：菜单栏出现低透明度灰灯 + `circle` 图标
2. **检测到 Claude Code**：Spring 动画 → 绿灯 `play.fill`
3. **需要确认**：Pulse 动画 + 红灯 + 声音提醒
4. **点击红灯**：展开菜单（Phase 1）或 Popover（Phase 2），显示等待原因
5. **无 Claude Code**：菜单项 "No Claude Code sessions detected — start Claude Code to begin"
